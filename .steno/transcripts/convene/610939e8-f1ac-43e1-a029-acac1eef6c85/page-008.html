## Critical Architecture Patterns

### Database & RLS
- **CRITICAL**: Avoid circular RLS dependencies (causes infinite recursion)
- Use `SECURITY DEFINER` functions with `SET search_path = ''` to prevent injection
- Service layer must accept authenticated Supabase clients
- User email denormalized to `profiles.email` (synced from `auth.users` via trigger)
- User creation: Pass role in `user_metadata` for `handle_new_user()` trigger

### UI Components
- **Charts**: Always use `ChartContainer` + `ChartConfig` (not raw Recharts)
- Avoid horizontal bar charts - use vertical bars with angled labels
- Convert counts to numbers: `Number(item.count) || 0`
- Certificate templates use flat structure: `background_color` not `background.color`

### Security & Services
- Security migrations for schema changes
- **Email**: Supabase for auth, Resend for operational emails
- **BotID Impact**: Call services directly in server actions (not `fetch()`)
- Security alerts have 30-min cooldown, send via Resend
- Rate limiting: Upstash Redis with tiered limits (Export: 10/hr, Admin: 200/15min, Global: 100/15min)

### Auth Flow Patterns
- **Supabase invites/magic links**: Use hash tokens (`#access_token`), client-side only
  - Must call `supabase.auth.setSession()` on client (not server callback)
  - Redirect to client page (e.g., `/auth/accept-invite`), not `/auth/callback`
- **OAuth account linking**: Use `linkIdentity()` to add OAuth to invited accounts
- **Redirect URLs**: Include both www and non-www in Supabase allowlist
- **Application Access Tokens**: Time-limited (7-day) SHA-256 tokens for email magic links
  - Generated via `createApplicationAccessToken()` in `lib/access-tokens.ts`
  - Validated in routes using `useAccessToken()` function

### Activities System (December 2025)
- **Public Play URL**: `/play?code=XXXXXX` - 6-character join codes
- **Anonymous support**: `allow_anonymous` flag per activity, participants stored in `activity_participants`
- **Progress tracking**: JSONB in `activity_progress` with win detection for Bingo
- **Leaderboard**: `get_activity_leaderboard()` RPC function returns `{ activity, leaderboard }`
- **Server-side redirects**: Play page uses server component to redirect before client hydration
- **API patterns**: Accept participant ID from `X-Participant-Id` header or query param

### Play Page Architecture
The `/play` page uses a **server component** to handle join codes:
1. Server reads `?code=` from searchParams
2. Queries Supabase with service role client
3. Redirects to `/play/[activityId]/join` before client JS runs

### API Flexibility
- Activity lookup accepts both `?code=` and `?id=` parameters
- Progress API accepts participant ID from `X-Participant-Id` header OR `?participant=` query param
- Leaderboard API returns `{ activity: {...}, leaderboard: [...] }`

### Bingo Win Detection
Progress POST endpoint automatically:
1. Checks marked items against win patterns (row, column, diagonal, corners, full)
2. Updates `has_bingo` and `winning_pattern` in progress data
3. Sets `completed_at` timestamp on first win
